Testing Strategies

人机混合代码库的测试策略
Article 0.3 of 20 ~17 min read 3 Source Books × 4 AI Angles

01 Core Insight — The Thesis

Testing has always been the discipline that turns code from "it seems to work" into "we can prove it works." Kent Beck's red-green-refactor cycle, Freeman and Pryce's outside-in acceptance testing, and Khorikov's emphasis on testing observable behavior over implementation details — these ideas built the trust infrastructure that makes professional software possible. Without tests, every change is a gamble.

测试一直是将代码从"似乎能工作"转变为"我们能证明它工作"的学科。Beck的红-绿-重构循环、Freeman和Pryce的由外而内验收测试、Khorikov对测试可观察行为（而非实现细节）的强调——这些理念构建了专业软件赖以存在的信任基础设施。

Here is what the AI era makes urgent: when a significant portion of your code is generated by AI, testing is no longer optional or secondary — it becomes the primary trust mechanism. AI can produce hundreds of lines of plausible-looking code in seconds. Without rigorous testing, that speed is not productivity — it is faster bug production. The classical testing books anticipated this moment without knowing it: Beck's insight that "the test is the specification" becomes literal when the test is the prompt that tells the AI what to build. Khorikov's rule — test behavior, not implementation — becomes essential when the implementation was written by a machine whose internal reasoning you cannot inspect.

AI时代使之紧迫：当代码库的相当部分由AI生成时，测试不再是可选的或次要的——它成为首要的信任机制。AI能在几秒内生成数百行看似合理的代码。没有严格测试，这种速度不是生产力，而是更快的缺陷生产。Beck的洞察"测试即规约"在AI时代变成了字面意义。

The Reframing 核心重构

Tests are the new code review. In a human-only codebase, you trust code because a human wrote it and another human reviewed it. In a human+AI codebase, you trust code because tests prove it works. The test suite becomes the primary artifact that humans review — it is easier to verify that a test correctly specifies behavior than to verify that an AI-generated implementation is correct line by line. TDD's "write the test first" was always good practice. Now it is the essential workflow: human writes the test (specification), AI generates the implementation, tests verify correctness.

测试是新的代码审查。在纯人类代码库中，你信任代码因为人写的、人审查的。在人机混合代码库中，你信任代码因为测试证明它有效。测试套件成为人类审查的首要工件——验证测试是否正确指定了行为，比逐行验证AI生成的实现要容易得多。TDD的"先写测试"从良好实践变为核心工作流。

02 Classical Foundation — Source Books

Three books define the modern testing canon. Beck gave us the discipline of test-first development. Freeman and Pryce showed how tests can drive architecture from the outside in. Khorikov provided the modern synthesis that resolves decades of debate about what to mock, what to test, and where the boundaries should be. Together, they form a complete framework for building trust through testing — a framework that the AI era demands we apply with even greater rigor.

三本书定义了现代测试经典。Beck赋予我们测试先行的纪律，Freeman和Pryce展示了测试如何由外而内驱动架构，Khorikov提供了现代综合方案——解决了关于mock什么、测试什么、边界在哪的数十年争论。它们共同构成通过测试建立信任的完整框架。

Test-Driven Development: By Example
Kent Beck · 2002
The red-green-refactor cycle that changed how professionals write software. Tests are not verification after the fact — they are a design tool. Writing the test first forces you to think about the interface before the implementation. In the AI era, this becomes: the test is the specification that tells the AI what to build.
FOUNDATION · TDD DISCIPLINE
Growing Object-Oriented Software, Guided by Tests
Freeman & Pryce · 2009
Outside-in TDD: start with a failing acceptance test that describes the feature from the user's perspective, then drive down through unit tests. The "walking skeleton" — a thin slice through the entire architecture — proves the system hangs together before fleshing out details. Mock-based testing defines collaboration contracts at boundaries.
FOUNDATION · OUTSIDE-IN TDD
Unit Testing: Principles, Practices, and Patterns
Vladimir Khorikov · 2020
The modern synthesis. Distinguishes output-based, state-based, and communication-based testing. His central argument: test observable behavior, not implementation details. Resist over-mocking. This principle is critical for AI-generated code — you cannot review the AI's implementation line by line, but you can verify its observable behavior through well-designed tests.
PRACTICE · MODERN SYNTHESIS

What connects these three books is a shared understanding that tests serve dual purpose: they verify correctness and they document intent. Beck's failing test says "this is what I want." Freeman and Pryce's acceptance test says "this is what the user needs." Khorikov's behavior-focused test says "this is what the system promises." When AI enters the picture, this dual purpose becomes triple: tests verify, document, and specify. The test suite becomes the contract between human intent and machine implementation — the source of truth that both sides must agree on.

三本书的共同认知：测试具有双重目的——验证正确性并记录意图。当AI加入时，双重目的变为三重：测试验证、记录、并规约。测试套件成为人类意图与机器实现之间的契约——双方必须达成一致的真相来源。

03 AI Co-Evolution — Four Angles

Each angle below takes one classical testing principle and extends it with an AI capability that transforms the testing workflow. The pattern: classical principle provides the what and why; AI provides the how much and how fast. The human remains the arbiter of what matters.

以下每个角度取一个经典测试原则，用AI能力加以扩展。经典原则提供"做什么"和"为什么"；AI提供"多少"和"多快"。人类仍然是"什么重要"的仲裁者。

Angle A · AI-Generated Test Suites — Who Tests the AI's Code? · AI生成的测试套件

Classical principle: Beck's TDD — write the test first, then the code. The failing test is the specification. The discipline of red-green-refactor ensures that every line of production code exists to satisfy a test. The test is not an afterthought; it is the driving force.

AI extension: When AI generates code, TDD's "test first" becomes the essential workflow rather than a nice-to-have discipline. The human writes a failing test that precisely specifies the desired behavior — this test is, in effect, the prompt. The AI generates an implementation to make it pass (green). Then the AI proposes additional edge-case tests that the human may not have considered: boundary values, null inputs, concurrency scenarios, error paths. The human reviews these AI-proposed tests (much easier to review than implementation code), approves or adjusts them, and now has a comprehensive test suite that the human+AI pair authored together. This is TDD's ultimate form — tests as specification, elevated from practice to necessity. 

# STEP 1: Human writes the failing test (RED) — this IS the spec
def test_discount_calculator_applies_bulk_discount():
    calc = DiscountCalculator(rules=load_rules("standard-2024"))

    result = calc.calculate(
        items=[Item("widget", price=100, qty=25)],
        customer_tier="gold"
    )

    assert result.subtotal == 2500
    assert result.discount_pct == 15       # bulk ≥20 units = 15%
    assert result.final_total == 2125

# STEP 2: AI generates implementation to pass this test (GREEN)

经典原则：Beck的TDD——先写测试，再写代码。AI延伸：人类写失败的测试（即规约），AI生成实现使之通过，然后AI提出额外的边界测试——人类审查这些测试（比审查实现代码容易得多），批准或调整。这是TDD的终极形态——测试即规约，从实践升级为必需。

Angle B · Mutation Testing Renaissance · 变异测试的复兴

Classical principle: Code coverage metrics (line coverage, branch coverage) are notoriously misleading. A test suite with 80% line coverage might still miss critical bugs because the tests execute the code without meaningfully asserting on its behavior. You can achieve 100% coverage with tests that assert nothing. The real question is not "how much code do the tests execute?" but "how many bugs would the tests catch?"

AI extension: Mutation testing answers that question directly. It works by injecting small faults ("mutations") into the production code — changing a > to >=, replacing a return value, removing a null check — and then running the test suite. If a test fails, the mutant is "killed" (good). If all tests pass despite the mutation, the mutant "survives" (bad — it means the test suite has a gap). Mutation testing has always been the gold standard for test quality, but it was prohibitively expensive: generating all possible mutations and running the full test suite for each one could take hours. AI makes mutation testing practical by generating targeted semantic mutations — not random syntactic changes, but meaningful fault injections based on understanding the code's logic. AI also analyzes surviving mutants and suggests exactly which test to add to kill them.

Level 1 — No Coverage Metrics
"It works on my machine." No test discipline. Bugs found in production. Each deployment is an act of faith.
Level 2 — Line Coverage Gate (e.g. 80%)
CI blocks PRs below coverage threshold. Better than nothing, but teams learn to game it — tests that execute code without meaningful assertions.
Level 3 — Branch Coverage + Assertion Density
Deeper metrics: branch coverage, assertions-per-test, mutation score for critical modules. Manual mutation testing tools (PIT, mutmut) run nightly.
Level 4 — AI-Targeted Mutation Testing
AI generates semantic mutations based on code intent, not random syntax changes. Mutation space shrinks 10x. Practical for every PR, not just nightly runs.
Level 5 — AI Mutation + Auto-Fix (2025→)
Surviving mutants auto-analyzed. AI suggests the exact test to add to kill each mutant. Developer reviews and approves. Test gaps close continuously.
Level 6 — Continuous Mutation Score (2026→)
Mutation score tracked per module, per sprint. AI maintains a living test-quality dashboard. "Mutation score dropped below 85% in payments module" triggers auto-generated test PRs.

经典原则：代码覆盖率指标具有误导性——80%行覆盖率不意味着能捕获80%的缺陷。变异测试是测试质量的金标准，但过于昂贵。AI延伸：AI生成有针对性的语义变异（非随机语法变更），变异空间缩小10倍，使变异测试在每个PR上都可行。AI分析存活的变异体并建议添加哪些测试来消灭它们。

Angle C · Contract-First Testing + AI Verification · 契约优先测试

Classical principle: Freeman and Pryce's outside-in approach starts with an acceptance test that describes the feature from the outside — what the user sees, what the API returns. This acceptance test drives the internal design. Khorikov reinforces this: test observable behavior at system boundaries, not internal implementation details. The boundary is the contract.

AI extension: Modern API-first development already produces machine-readable contracts — OpenAPI specs for REST, AsyncAPI specs for event-driven systems, GraphQL schemas, Protobuf definitions. These contracts are the "acceptance test specifications" that Freeman and Pryce envisioned, but in a format that AI can consume directly. AI can generate comprehensive conformance test suites from API specs — testing every endpoint, every parameter combination, every error code documented in the spec. More critically, AI can detect contract drift: cases where the implementation has diverged from the spec (a field was added without updating the OpenAPI doc, a status code changed, a required field became optional). This turns the spec from a static document into a living, continuously verified contract.

Concrete example: A team maintains an OpenAPI 3.1 spec as the source of truth for their payment service API. In CI, an AI agent reads the spec, generates contract tests for every documented endpoint (including error scenarios, rate limits, and pagination edge cases), runs them against the actual service, and flags any discrepancies. When a developer changes the API without updating the spec, the drift detection surfaces it within minutes — not weeks later when a downstream consumer breaks.

经典原则：Freeman和Pryce的由外而内方法从描述功能的验收测试开始。Khorikov强化：在系统边界测试可观察行为。AI延伸：OpenAPI/AsyncAPI/GraphQL契约是机器可读的"验收测试规约"。AI从API规约生成全面的一致性测试套件，更关键的是，AI检测契约漂移——实现偏离规约的情况。将规约从静态文档变为持续验证的活契约。

Angle D · Building Trust in AI-Generated Code · 构建对AI生成代码的信任

Classical principle: Trust in code has always rested on two pillars: code review (a human reads it and says "this looks right") and testing (automated verification that it does what it should). Khorikov's "test observable behavior, not implementation details" is the most practical formulation of the testing pillar — you don't need to understand how the code works internally, only that it produces the correct outputs for given inputs.

AI extension: AI-generated code needs a higher trust bar than human-written code, because the author (the AI) cannot explain its reasoning the way a human colleague can. The four-layer trust framework addresses this systematically. Layer 1: the human writes the test first — this is the specification, the non-negotiable contract. Layer 2: AI generates the implementation to pass the test, plus proposes additional tests. Layer 3: the human reviews the tests first (not the implementation) — reviewing tests is cognitively easier because tests describe what the code should do, not how it does it. Layer 4: mutation testing validates test quality — if the tests can survive code mutations, the test suite itself has gaps. Only after all four layers pass does the code earn full trust. The key insight: the test suite is the primary artifact that humans review, because tests are a more efficient compression of the code's intended behavior than the implementation itself.

Concrete example: A team adopts the trust pipeline for a new payment processing module. Developer writes 12 failing tests specifying all discount rules, tax calculations, and error handling. AI generates the implementation (340 lines). Instead of reviewing 340 lines of implementation, the tech lead reviews 12 test specifications (80 lines) — confirming they capture all business rules. Then AI mutation testing runs 45 targeted mutations against the implementation; 43 are killed, 2 survive. AI suggests two additional assertions to cover the gaps. After adding them, the mutation score hits 100%. The team ships with confidence — not because they reviewed every line, but because every behavior is tested and every test is validated by mutation analysis.

经典原则：代码信任建立在代码审查和测试两个支柱上。Khorikov的"测试可观察行为"是测试支柱的最实用表述。AI延伸：AI生成的代码需要更高的信任门槛。四层信任框架：(1)人类先写测试，(2)AI生成实现+额外测试，(3)人类先审查测试（比审查实现容易），(4)变异测试验证测试质量。关键洞察：测试套件是人类审查的首要工件。

05 Practical Checklist — Building Your AI-Augmented Testing Practice

To move from ad-hoc testing to a rigorous, AI-augmented trust pipeline, follow this staged checklist. Each stage builds on the previous — the AI-powered stages only work if the classical foundations are solid.

从临时测试迈向严格的AI增强信任管线的分阶段检查清单。每个阶段建立在前一阶段之上——AI驱动的阶段只有在经典基础扎实时才有效。

S2 Establish minimum test coverage gate in CI (line + branch)
S2 Standardize test framework and naming conventions across repos
S3 Adopt TDD for all new code — team reads Beck Ch. 1-8
S3 Write integration tests for critical paths (walking skeleton)
S3 Triage and fix flaky tests — zero-flake CI as team policy
S4 AI generates edge-case tests for every new PR on changed files
S4 Enable mutation testing in CI for critical modules (payments, auth)
S5 Contract-first testing from OpenAPI/AsyncAPI specs in CI pipeline
S5 AI contract drift detection — alert on spec/implementation divergence
S6 Full AI trust pipeline: human spec → AI implement → review tests → mutate
S6 AI-maintained test suite health dashboard (mutation score, flake rate, drift)
S6 AI auto-generates test PRs when mutation score drops below threshold

06 The Architect's Takeaway

The three books on this article's shelf — Beck, Freeman and Pryce, Khorikov — gave us everything we need to test well. What they couldn't anticipate was the scenario where a significant fraction of production code is written by a machine. In that scenario, testing transforms from a quality practice into the primary trust infrastructure. Without testing discipline, AI code generation is just faster bug production — plausible-looking code that nobody has verified, shipped at a pace that outstrips any team's review capacity.

三本书给了我们测试所需的一切。它们无法预见的是生产代码大量由机器编写的场景。在该场景下，测试从质量实践变为首要的信任基础设施。没有测试纪律，AI代码生成不过是更快的缺陷生产。

The trust pipeline this article proposes — human specification, AI implementation, human test review, mutation validation — is not a theoretical framework. It is a concrete workflow that teams can adopt incrementally, starting with "human writes the test, AI generates the code" (Stage 4) and building toward full mutation-validated trust (Stage 6). The architect's role in this pipeline is to design the test strategy, set the quality gates, and decide which modules deserve which level of rigor. Not every CRUD endpoint needs mutation testing. But your payment processing, your authentication flow, your data pipeline transformations — those need the full trust pipeline. The judgment of where to invest testing rigor is irreducibly human, and it is the architect's most consequential decision in the AI era.

本文提出的信任管线——人类规约、AI实现、人类审查测试、变异验证——不是理论框架，而是团队可以渐进采纳的具体工作流。架构师的角色是设计测试策略、设置质量门禁、决定哪些模块值得哪种严格程度。"在哪里投入测试严格性"的判断是不可还原的人类决策，也是架构师在AI时代最重要的决定。
