AI-Era Refactoring

AI时代的重构：机器提议，人类决策
Article 0.2 of 20 ~18 min read 4 Source Books × 4 AI Angles

01 Core Insight — The Thesis

Refactoring has always been the software engineer's quiet discipline — the daily act of improving code structure without changing its external behavior. Martin Fowler codified the catalog. Michael Feathers taught us how to get legacy code under test. Kent Beck reminded us to tidy first, before adding features. Adam Tornhill showed that git history reveals where the real pain lives. These ideas are battle-tested and foundational.

重构一直是软件工程师的静默修行——在不改变外部行为的前提下，日常改善代码结构。Fowler编目了重构手法，Feathers教我们如何让遗留代码进入测试，Beck提醒我们先整理再添加功能，Tornhill证明git历史揭示了真正的痛点所在。

Here is what changes in the AI era: refactoring shifts from a human-scheduled activity to a continuously detected, AI-proposed, human-approved workflow. The bottleneck was never knowing how to refactor — Fowler's catalog covers that. The bottleneck was knowing where to refactor (which files hurt the most?), when to refactor (is now the right time?), and whether it's safe (do we have enough tests?). AI addresses all three bottlenecks simultaneously.

AI时代的变化：重构从人工安排的活动转变为持续检测、AI提议、人类批准的工作流。瓶颈从来不是"如何"重构——Fowler的目录已覆盖。瓶颈是"在哪"重构、"何时"重构、以及"是否安全"——AI同时解决这三个瓶颈。

The Reframing 核心重构

Machines propose, humans dispose. AI analyzes git history to find hotspots, generates characterization tests for untested legacy code, and proposes refactoring sequences from Fowler's catalog. The human architect's role shifts from "doing the refactoring" to "approving the refactoring plan, verifying the safety net, and steering the sequence." This is not less skill — it is different skill. The judgment of what matters remains irreducibly human.

机器提议，人类决策。AI分析git历史找到热点，为未测试的遗留代码生成特征测试，并从Fowler目录中提议重构序列。架构师的角色从"执行重构"转变为"批准重构计划、验证安全网、引导序列"。这不是更少的技能，而是不同的技能。

02 Classical Foundation — Source Books

Four books form the intellectual backbone of modern refactoring practice. Each addresses a different dimension: Fowler gives you the vocabulary, Feathers gives you the safety techniques, Beck gives you the timing philosophy, and Tornhill gives you the forensic evidence. Together, they form a complete refactoring toolkit that AI extends but never replaces.

四本书构成现代重构实践的知识骨架。Fowler提供词汇表，Feathers提供安全技术，Beck提供时机哲学，Tornhill提供取证证据。它们共同构成完整的重构工具包——AI扩展但永远不会取代。

Refactoring, 2nd Edition
Martin Fowler · 2018
The canonical catalog of refactoring patterns. Extract Method, Move Field, Replace Conditional with Polymorphism — each with mechanical steps that are precise enough for an AI agent to execute. Fowler's key insight: refactoring is not rewriting; it's a sequence of behavior-preserving transformations.
FOUNDATION · THE CATALOG
Working Effectively with Legacy Code
Michael Feathers · 2004
The survival guide for code without tests. Feathers' "seams" concept — points where you can alter behavior without editing production code — is the key to making untestable code testable. His characterization tests capture current behavior as a safety net before any change.
FOUNDATION · LEGACY SAFETY
Tidy First?
Kent Beck · 2023
Beck's answer to "when do I refactor?" is elegantly simple: tidy first, then change behavior. Small structural tidyings — guard clauses, extract helpers, normalize symmetries — before each feature. This micro-refactoring philosophy maps perfectly to AI-proposed pre-change cleanups.
PRACTICE · TIMING
Your Code as a Crime Scene
Adam Tornhill · 2015
Forensic analysis of codebases using git history. Hotspot detection: files with high change frequency AND high complexity are your top refactoring targets. Temporal coupling analysis reveals hidden dependencies. The data-driven approach turns "I think this file is bad" into "the evidence shows this file costs us."
PRACTICE · FORENSICS

What connects these four books is a shared recognition that refactoring is not a luxury — it is an economic decision. Fowler frames it as paying down design debt. Feathers frames it as making change possible. Beck frames it as an investment that pays off immediately (the next feature is cheaper). Tornhill frames it as risk reduction backed by data. The AI era amplifies each of these frames: AI makes identification cheaper (Tornhill), safety cheaper (Feathers), timing clearer (Beck), and execution faster (Fowler). The human role shifts from execution to judgment — deciding which refactoring to approve, how far to go, and when to stop.

四本书的共同认知：重构不是奢侈品，而是经济决策。Fowler视其为偿还设计债务，Feathers视其为使变更成为可能，Beck视其为立即回报的投资，Tornhill视其为数据支撑的风险降低。AI时代放大了每一个维度——人类角色从执行转向判断。

03 AI Co-Evolution — Four Angles

Each angle below takes one classical refactoring principle and extends it with an AI capability that was not possible five years ago. The pattern is always the same: the classical principle provides the what and why; AI provides the how fast and how safely.

以下每个角度取一个经典重构原则，并用五年前不可能的AI能力加以扩展。模式始终相同：经典原则提供"做什么"和"为什么"；AI提供"多快"和"多安全"。

Angle A · AI-Powered Hotspot Detection — AI驱动的热点检测

Classical principle: Tornhill's behavioral code analysis mines git history to find "hotspots" — files that change frequently AND have high complexity. These are the files where bugs cluster, where onboarding takes longest, and where refactoring has the highest ROI. The key metric is change frequency × cyclomatic complexity.

AI extension: Traditional hotspot detection uses static metrics. AI adds a semantic layer: it can read each commit message, understand the intent of changes, and classify whether changes are bug fixes, features, or refactoring. This means AI can distinguish a file that changes often because it's a healthy extension point from a file that changes often because it's a maintenance nightmare. Furthermore, AI can correlate hotspots with incident reports, on-call pages, and deployment rollbacks — producing a prioritized refactoring backlog ranked by actual business impact, not just code metrics.

Concrete example: A nightly CI job runs git log --numstat over the past 90 days, feeds the output to an AI agent along with complexity metrics from SonarQube, and produces a ranked "Refactoring Backlog" in the project wiki. Each entry includes: file path, hotspot score, classification (bug-magnet / coupling-hub / complexity-trap), suggested Fowler refactoring patterns, and estimated risk. The team reviews this list every sprint planning — turning "we should refactor sometime" into "here's the data-driven priority list."

经典原则：Tornhill的行为代码分析挖掘git历史找到"热点"——变更频繁且复杂度高的文件。AI延伸：AI添加语义层——理解提交意图、区分健康扩展点与维护噩梦、关联事故报告产出按业务影响排序的重构待办列表。具体示例：夜间CI作业运行git log + SonarQube，AI生成排序的"重构待办列表"。

Angle B · AI-Generated Characterization Tests — AI生成的特征测试

Classical principle: Feathers' central insight is that before you can refactor legacy code, you need a safety net. His "characterization tests" capture the current behavior of code — not what it should do, but what it actually does. You write tests, observe the output, and pin it as the expected behavior. Only then is it safe to refactor.

AI extension: Writing characterization tests manually is tedious and time-consuming — which is why it rarely gets done. AI changes this equation dramatically. An AI agent can read a legacy method, trace its dependencies, identify Feathers' "seams" (constructor injection points, interface boundaries, preprocessor seams), and generate a comprehensive characterization test suite in minutes. The AI doesn't need to understand the business intent — it only needs to capture current behavior, which is exactly what LLMs excel at: reading code and producing matching test assertions. 

// Legacy code: untested 300-line method in InvoiceCalculator
// AI identifies the "object seam" (Feathers Ch.4): inject TaxService

// ✓ AI-generated characterization test — pins current behavior
@Test
void characterize_calculateTotal_withChileanTax() {
    TaxService taxService = new ChileanSIITaxService();
    InvoiceCalculator calc = new InvoiceCalculator(taxService);

    Invoice invoice = buildTestInvoice(
        lineItem("Widget", 100, 3),   // qty=3, price=100
        lineItem("Gadget", 250, 1)    // qty=1, price=250
    );

    InvoiceResult result = calc.calculateTotal(invoice);

    // Pinned from actual execution — NOT from requirements
    assertThat(result.getSubtotal()).isEqualTo(550.00);
    assertThat(result.getTaxAmount()).isEqualTo(104.50);  // 19% IVA
    assertThat(result.getTotal()).isEqualTo(654.50);
}

// AI generates 15-20 such tests covering edge cases:
// empty invoice, zero-quantity lines, rounding boundaries,
// exempt items, multi-currency — all pinned from actual output

经典原则：Feathers的核心洞察——重构前需要安全网，特征测试捕获代码当前行为。AI延伸：AI读取遗留方法、识别"缝隙"（构造函数注入、接口边界）、在几分钟内生成全面的特征测试套件。AI不需要理解业务意图——只需捕获当前行为。

Angle C · Strangler Fig Accelerated by AI — AI加速的绞杀者模式

Classical principle: Fowler's Strangler Fig pattern (named after the tropical tree that gradually envelops its host) is the safest approach to large-scale legacy migration. Instead of a Big Bang rewrite, you route new functionality through a new system while the old system gradually loses responsibilities. The pattern requires a routing layer (façade) that decides which system handles each request.

AI extension: The Strangler Fig pattern is safe but slow — each migration step requires understanding both the old and new system's behavior. AI accelerates every step: (1) AI reads legacy code and generates a behavioral specification of the module being strangled; (2) AI scaffolds the new implementation from that spec, following the project's current conventions (read from .ai/conventions.md); (3) AI generates comparison tests that verify old and new produce identical output for the same input; (4) AI monitors the routing layer in production to detect behavioral divergence. What used to take a team three months per module can now take three weeks — with higher confidence.

Concrete example: Migrating a legacy invoice formatting module from a monolithic Java service to a new bounded context. AI reads the old InvoiceFormatter.java (2,400 lines), generates a behavioral spec documenting all formatting rules, scaffolds a new InvoiceFormattingService following hexagonal architecture conventions, and produces a shadow-mode comparison test suite that runs both old and new in parallel, asserting byte-for-byte PDF output identity.

经典原则：Fowler的绞杀者模式——用新系统逐步替代旧系统，而非大爆炸式重写。AI延伸：AI加速每一步——生成行为规范、按项目约定搭建新实现、生成对比测试、监控路由层检测行为偏差。过去团队三个月/模块的工作现在可以三周完成。

Angle D · Continuous Refactoring Signals in CI/CD — CI/CD中的持续重构信号

Classical principle: Fowler advocates for "continuous refactoring" — refactoring as part of every development task, not a separate phase. Beck's "tidy first" crystallizes this: before you add a feature, tidy the code you're about to touch. The problem is that without signals, developers forget. The refactoring intention decays under delivery pressure.

AI extension: CI/CD pipelines can now embed AI-powered "refactoring signal" checks. On every PR, an AI agent: (1) runs hotspot analysis on the files touched by the PR; (2) checks if any touched file is on the "refactoring backlog"; (3) if so, suggests a "tidy first" refactoring that should precede the feature change; (4) estimates the cost of not refactoring (increased complexity, coupling, future bug probability). This turns Beck's philosophy into a concrete, automated nudge. The AI doesn't block the PR — it informs the developer and the reviewer, making the refactoring-vs-ship trade-off explicit and data-backed.

Level 1 — No Refactoring Culture
"We'll refactor when we have time." Tech debt accumulates silently. No visibility into what's degrading.
Level 2 — Periodic Tech Debt Sprints
Quarterly "cleanup sprints." Some improvement, but targets are chosen by gut feel. Gains erode quickly.
Level 3 — SonarQube Gates
Static analysis blocks new code smells. Prevents degradation but doesn't improve existing code.
Level 4 — Data-Driven Hotspots (Tornhill)
Git-history analysis identifies top hotspots. Team tackles highest-ROI files first. Progress is measurable.
Level 5 — AI Refactoring Signals (2025→)
Every PR gets an AI "tidy first" suggestion on touched hotspot files. Refactoring cost/benefit is explicit. Developers are nudged, not blocked.
Level 6 — Autonomous Refactoring PRs (2026→)
AI opens refactoring PRs with characterization tests, diff, and impact analysis. Human reviews and merges. Refactoring becomes continuous, not episodic.

经典原则：Fowler倡导"持续重构"——Beck的"先整理"将其具体化。问题是没有信号，开发者就会遗忘。AI延伸：CI/CD管道嵌入AI重构信号——每个PR的AI代理分析触及的文件、检查是否在重构待办列表中、建议"先整理"重构、估算不重构的成本。将Beck的哲学变成具体的、自动化的提示。

06 The Architect's Takeaway

The four books on this article's shelf — Fowler, Feathers, Beck, Tornhill — gave us everything we need to refactor well. What they couldn't give us was the capacity to refactor continuously at scale. AI fills that gap. It detects hotspots before humans notice them, generates safety nets for code that no one dared to touch, accelerates migration patterns that used to take quarters, and embeds refactoring awareness into the daily PR flow.

四本书给了我们重构所需的一切。它们无法给予的是大规模持续重构的能力。AI填补了这一空白——在人类注意到之前检测热点、为无人敢碰的代码生成安全网、加速需要数季度的迁移模式、将重构意识嵌入日常PR流程。

But the architect's judgment remains the irreducible core. AI can tell you that InvoiceCalculator.java is hotspot #1 with 47 commits in 90 days and cyclomatic complexity of 83. It cannot tell you whether now — mid-sprint, pre-launch — is the right moment to refactor it. It cannot weigh the organizational cost of a multi-week migration against the opportunity cost of shipping a new feature. That calculus is human, contextual, and political. The architect who masters both the classical refactoring discipline and the AI-augmented workflow doesn't just write better code — they lead teams that continuously improve their codebase as a strategic asset.

但架构师的判断力仍然是不可还原的核心。AI能告诉你某文件是热点#1，它无法告诉你在冲刺中期、发布前夜是否是重构的正确时机。掌握经典重构纪律和AI增强工作流的架构师不仅写出更好的代码——他们带领团队持续改善代码库，使其成为战略资产。
